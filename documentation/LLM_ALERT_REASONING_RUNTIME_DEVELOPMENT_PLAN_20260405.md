# LLM Alert Reasoning Runtime Development Plan / LLM 告警推理运行时开发计划（2026-04-05）

这份文档专门回答两个问题，第一是从当前仓库状态出发，这个 LLM runtime 应该如何按最小可运行闭环逐步开发，避免一上来就为了“先进”而做大而全重构，第二是当全部阶段完成后，系统最终应该具备哪些具体功能、哪些功能必须明确保持在审批边界之外、哪些能力适合被写进论文对比实验、哪些能力适合被写进简历亮点。当前仓库的实际基础并不差，甚至可以说已经具备了非常适合发展成 agentic runtime 的骨架，因为 deterministic plane 已经具备 raw fact、deterministic alert、dual persistence、suggestion output、runtime projection 这些关键阶段，`core/aiops_agent` 已经有 alert-scope 和 cluster-scope 两种入口，ClickHouse history lookup 已经存在，cluster aggregator 已经提供 same-key repeated pattern 的确定性聚合入口，这意味着后续开发并不是从零起步，而是把现有 bounded suggestion layer 升级成 traceable reasoning runtime。考虑到你强调论文价值、简历价值、审计价值和工程真实性，开发顺序必须始终坚持四个原则，第一是 deterministic first, agentic second，任何阶段都不能让模型回流去替代 correlator 的成立性判断；第二是结构化优先，所有中间对象都要先定义 schema，再考虑自然语言展示；第三是 review 和 fallback 必须早于自动执行，哪怕 runbook 先只生成给人看，也必须把风险、审批和 rollback 结构做出来；第四是先打通最小闭环，再扩图、再扩检索、再扩模型、再扩 control boundary，不做一次性大重构。基于这些原则，第一阶段应该做的是“runtime seed 和对象骨架落地”，这一阶段的目标并不是让模型变强，而是让系统先拥有一套未来可以承接更深 reasoning 的对象边界。从当前代码出发，这一阶段已经开始落地的内容包括把 `alert_reasoning_runtime` 子目录放进 `core/aiops_agent` 下，用高可读命名方式放入 `candidate_event_graph.py`、`investigation_session.py`、`reasoning_trace.py`、`runbook_plan_outline.py` 和 `rule_based_seed_adapter.py`，并通过 `evidence_bundle.py` 把这些 seed 接到当前 alert-scope 和 cluster-scope pipeline 中。接下来这一阶段还应继续补完两类工作，一类是 schema 稳定化，例如把 candidate event graph 中的 node type、relation type、basis、deterministic score、evidence refs 定义成更严格的枚举或 schema 文档，把 investigation session 中的 working memory seed 字段固定下来，把 runbook outline 的 prechecks、operator actions、approval boundary、rollback guidance 做成正式 contract，另一类是 runtime projection 适配，即在 frontend gateway 和 runtime snapshot 中逐步显示 candidate graph id、session id、trace id、runbook outline summary，让 operator console 至少能感知这条新下游链已经存在。第一阶段完成后的功能非常明确：系统仍然和现在一样从已成立 alert 出发生成 suggestion，但 suggestion 不再只是 summary+hypotheses+actions，而会显式携带 reasoning runtime seed，为后续 deeper reasoning 留出对象边界，同时不改变当前 deterministic alert 链，也不改变 current provider 的 template/http 抽象，这就是一个很典型的最小可运行闭环。第二阶段应该做的是“Evidence Pack V2 和 graph-aware retrieval 落地”，这一阶段解决的是你非常在意的上下文质量问题。当前 evidence bundle 已经包含 topology、history、rule、path、policy、device、change 等字段，但对 future reasoning 来说还不够，因为它更像“上下文拼接结果”，而不是“可按任务阶段调度和审查的证据对象”。因此第二阶段需要将 evidence pack 正式升级成支持 direct evidence、supporting evidence、contradictory evidence、missing evidence、freshness、source reliability、lineage 的对象，并把现有 ClickHouse history lookup、未来 topology 邻域查询、change-window 检索、runbook applicability 检索统一视为 context service 的子能力。在代码层面，这一阶段建议新增 `evidence_pack_builder.py`、`graph_aware_retrieval.py`、`context_phase_router.py`、`working_memory_store.py`、`context_compression.py` 这类可读性强的模块名，而不是用抽象过度的缩写命名。这里的 graph-aware retrieval 不等于完整 GraphRAG，它的近期目标是根据当前 cluster 或 investigation session 的 selected nodes 和 relation edges 去找拓扑邻居、时间窗口重叠的变更记录、历史相似 pattern、runbook 候选，而不是从海量文档中做复杂的图索引。这个阶段完成后，系统应该能够做到对一个 alert 或 cluster 生成更严格的 typed evidence pack，并且在下一阶段推理时按 phase routing 精准给上下文，显著减少“把所有东西拼进一个 prompt 导致遗忘和答非所问”的问题。第三阶段应该做的是“Hypothesis Engine 和 Review Engine 落地”，这一阶段是真正把 suggestion generator 变成 reasoning runtime 的转折点。这里不建议直接上去中心化 swarm，而建议继续围绕 supervisor-style pipeline 或 LangGraph state machine 做，因为对于你当前仓库这种 alert downstream、审计导向、强边界系统来说，更重要的是有明确状态流转和回退路径，而不是 agent 数量。Hypothesis Engine 的目标不是给出一个自由文本结论，而是输出一个 `HypothesisSet`，其中每个 hypothesis 都要至少包含 hypothesis id、statement、support evidence refs、contradict evidence refs、missing evidence refs、confidence、next best evidence action。Review Engine 则是这个系统成熟度的关键所在，它需要固定检查 evidence sufficiency、temporal freshness、topology consistency、overreach risk、runbook executability、rollback readiness 等维度，并输出 `ReviewVerdict`，明确说明是通过、需要补证据、需要重做假设，还是因为越权或高风险停在 human gate。这个阶段在工程上的核心价值是把“模型直接给建议”改造成“先提假设，再批判，再决定是否能进入 runbook drafting”，这会大幅增强论文对比和简历表达的可信度，因为你可以非常清楚地展示不同阶段的对象、失败回退路径和性能边界。第四阶段应该做的是“Runbook retrieval 和 runbook planning 正式化”，也就是把你已经确认要做的 runbook 能力从 outline 升级成真正的 planning 输出。这一阶段需要新增 runbook 文档源，不论这些文档来自 SOP、历史 case、incident notes 还是人工整理的处置模板，都应该通过一个明确的 retrieval 层进入系统，而不是在 prompt 里硬编码。建议增加 `runbook_retriever.py`、`runbook_applicability_ranker.py`、`runbook_plan_builder.py` 和 `runbook_review.py` 这类模块，把 runbook 相关流程拆成四步：先检索候选 runbook，再根据当前 alert/cluster/evidence/hypothesis 进行适配性排序，再生成结构化 runbook plan，最后做 runbook review。最终的 `RunbookPlan` 对象应包含 applicability、prechecks、evidence refs、diagnostic steps、recommended actions、risk level、blast radius、approval required、rollback prerequisites、rollback steps、missing information、operator handoff 等内容。这一阶段的一个关键约束是，runbook 计划可以越来越具体，但在 control plane 未实现之前仍然不能直接自动执行，这条边界需要同时体现在文档、schema、frontend projection 和 review verdict 里。第五阶段应该做的是“Reasoning state machine 和 LangGraph harness”，这一阶段不是为了“跟热门框架”，而是为了把前面所有对象真正串起来。当前建议的状态流应当是 `SessionInit -> ClusterSelectOrRefine -> ContextAssemble -> EvidencePackBuild -> HypothesisGenerate -> HypothesisCritique -> RunbookRetrieve -> RunbookDraft -> RunbookReview -> SuggestionProjection`，并保留若干固定 fallback 边，例如 `cluster_too_broad -> ClusterSelectOrRefine` 表示聚合粒度不合适需要拆簇重来，`evidence_insufficient -> ContextAssemble` 表示当前证据不足需要补检索，`hypothesis_conflict -> HypothesisGenerate` 表示假设集存在冲突需要重建，`runbook_not_applicable -> RunbookRetrieve` 表示当前拿到的 runbook 候选不适合当前 cluster 或 hypothesis，`runbook_overreach -> RunbookDraft` 表示计划超越当前控制边界需要重写，`high_risk_or_missing_approval -> human_gate` 表示流程必须停在人工审批边界。选择 LangGraph 的意义主要在于它适合显式状态图、可恢复执行、带 checkpoint 的长任务流程，而不是它能神奇地解决领域建模问题，因此实现时一定要坚持领域对象自定义、状态机框架外置这个原则。第六阶段应该做的是“Tool contracts、dry-run diagnostics 和未来 control plane 预留”，这一阶段并不意味着进入自动修复，而是为以后可能的 governed control plane 铺路。你已经强调 approval、rollback、audit 边界必须清晰，所以近期更合适的能力不是直接写设备配置，而是增加一层只读或 dry-run 的 diagnostic execution layer，例如 ping、traceroute、DNS lookup、policy diff、route check、config parse、path compare、runbook precheck runner 等，这些能力应通过统一工具契约接入 reasoning runtime，并带有 allowlist、参数校验、超时、结果结构化和命令审计。这个阶段最合适的表达方式不是“实现自动运维”，而是“实现了受控诊断执行器和未来审批写回前的工具边界”，论文和简历上都会更可信。第七阶段应该做的是“Tracing、replay 和 eval system”，这一步对论文尤其重要，但对简历也同样有价值。最终系统必须能够记录每个 investigation session 的 graph snapshot、evidence pack、hypothesis set、review verdict、runbook plan、tool call trace、suggestion projection，并支持 replay，这样你才能做真正有说服力的对比实验，例如 deterministic baseline、rule-based suggestion baseline、LLM enhanced evidence pack、LLM enhanced hypothesis without critique、LLM enhanced hypothesis with critique、LLM enhanced runbook planning with human gate 等不同策略的效果对比。如果没有这一层，后面很多研究叙述都只能停留在概念层。关于最终功能的完成态，可以从 operator 视角来描述：系统最终应支持从单个 alert 或 cluster trigger 启动 investigation session，自动生成 candidate event graph 并围绕 seed 节点扩图，按阶段装配 evidence pack，生成和批判 root cause hypotheses，检索和适配 runbook 候选，输出结构化 runbook plan，展示风险与审批边界，并把整个 reasoning 过程以可追溯、可回放、可审计的形式投影到 frontend runtime；同时系统还应支持未来在图中手动选择多个事件作为 seed，由 deterministic graph construction 和 graph-aware retrieval 先建立候选关系，再由 reasoning runtime 解释为什么这些事件属于同一 incident、为什么某些边是次要的、为什么某份 runbook 更适用。关于功能边界的完成态，也必须同样明确：系统最终不应让 LLM 替代 correlator 的 alert 判定，不应让模型直接读取原始日志洪流做告警成立判断，不应在没有人工审批和 rollback 契约的情况下直接自动执行 remediation write path，不应把自由文本当作一等控制对象，不应在没有 trace 和 evidence refs 的情况下把建议直接推给 operator。关于论文实验和简历价值，这条开发路线最大的优势在于每一阶段都天然形成可对比对象。第一阶段可以对比“无 reasoning seed suggestion”与“带 reasoning seed 的 projection”；第二阶段可以对比“普通上下文拼接”与“typed evidence pack + graph-aware retrieval”；第三阶段可以对比“single-pass hypothesis generation”与“hypothesis + critique loop”；第四阶段可以对比“recommended actions list”与“structured runbook plan”；第五阶段可以对比“linear pipeline”与“stateful workflow with fallback and human gate”；第六阶段可以对比“无工具约束建议”与“dry-run diagnostics + tool governance”；第七阶段可以评估 replayability、trace completeness、evidence coverage、runbook applicability 和 overreach rejection rate。为了便于执行，近期建议的实际开发顺序应该是先把第一阶段彻底做扎实，再继续第二阶段，而不是先引入 LangGraph 或 GraphRAG。更具体地说，下一轮开发最值得优先做的事包括：把当前 runtime seed 字段补上更严格的 schema 约束和测试；把 frontend runtime gateway 加上最小 projection 展示；设计 Evidence Pack V2 的结构化字段；实现第一版 graph-aware retrieval 接口但先用现有 ClickHouse/history/topology/change 数据占位；设计 HypothesisSet、ReviewVerdict 和 RunbookPlan 三个核心 schema；最后再决定哪些状态节点适合先接入 LangGraph harness。只要按这条顺序推进，系统在每个阶段都会有清晰的“已实现功能”和“边界内能力”，不会陷入长期大重构，也不会出现简历和论文只能写未来规划、不能写工程事实的问题。
