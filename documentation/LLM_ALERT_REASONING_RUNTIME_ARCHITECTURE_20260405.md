# LLM Alert Reasoning Runtime Architecture / LLM 告警推理运行时架构说明（2026-04-05）

本文档定义当前分支中的 LLM downstream path。起点是 `netops.alerts.v1`。输入对象是已成立的 alert contract 和 cluster trigger。输出对象是结构化 suggestion、候选事件图、调查会话、推理追踪和 runbook 计划。边缘解析、事实转发、规则判定、JSONL 审计和 ClickHouse 存储继续沿现有主链运行。当前代码已经把 `reasoning_runtime_seed` 挂到 `evidence_bundle`，并把 `candidate_event_graph_id`、`investigation_session_id`、`reasoning_trace_id` 和 `runbook_plan_outline` 投影到 suggestion。

> [!IMPORTANT]
> `core/correlator` 负责第一判定。`core/aiops_agent` 负责 alert downstream 推理与建议。

> [!WARNING]
> `192.168.1.23` 仅适合边缘采集与转发。该节点为 `Intel Xeon E3-1220 v5`、`4` 个 CPU 线程、`7.7 GiB` 内存、`914 GiB` 根分区。`192.168.1.27` 负责核心平面。该节点为 `Intel Xeon Silver 4310`、`24` 个 CPU 线程、`14 GiB` 内存、`1.8 TiB` 根分区。当前代码路径不在任一节点上启用本地 GPU 推理。

## 对象

当前实现放在 `core/aiops_agent/alert_reasoning_runtime`。这个目录存放代码，不存放运行中的数据。运行中的数据仍然落在 `/data/netops-runtime`。`candidate_event_graph.py` 负责从已成立 alert 或 cluster trigger 构造候选事件图。图节点当前覆盖 `alert_event`、`incident_cluster`、`service`、`device`、`alert_rule`、`network_path`、`change_record`、`historical_alert` 和 `historical_pattern`。图边当前覆盖 `targets_service`、`originates_from_device`、`emitted_by_rule`、`traverses_path`、`temporally_adjacent_to_change`、`resembles_recent_alert`、`contains_seed_alert`、`contains_related_alert` 和 `matches_repeating_cluster_pattern`。每条边都带 `basis`、`deterministic_score` 和 `evidence_refs`。

`investigation_session.py` 定义当前会话对象。会话对象记录 `session_scope`、`entrypoint`、`selected_node_ids`、`expanded_node_ids`、`current_stage`、`recommended_retrievers` 和 `working_memory_seed`。当前代码已经提供 alert 和 cluster 两种 seed session。`reasoning_trace.py` 定义 trace seed，字段包括 `trace_id`、`trace_scope`、`trace_status`、`completed_stages` 和 `pending_stages`。`runbook_plan_outline.py` 定义当前版本的 runbook 大纲，字段包括 `plan_scope`、`applicability`、`prechecks`、`operator_actions`、`approval_boundary` 和 `rollback_guidance`。`evidence_pack_v2.py` 已经进入代码。这个对象当前挂在 `evidence_bundle["evidence_pack_v2"]`。字段固定为 `schema_version`、`pack_id`、`pack_scope`、`group_order`、`entry_fields`、`alert_ref`、`direct_evidence`、`supporting_evidence`、`contradictory_evidence`、`missing_evidence`、`freshness`、`source_reliability`、`lineage` 和 `summary`。每条 evidence entry 当前固定为 `evidence_id`、`kind`、`status`、`label`、`value`、`source_section`、`source_field`、`source_ref` 和 `rationale`。这份对象现在是 hypothesis、review 和 runbook draft 的统一输入对象。`phase_context_router.py` 已经改为把 `evidence_pack_v2` 放入每个阶段的 payload，并按阶段裁剪 evidence group。当前阶段矩阵固定如下：`hypothesis_generate` 读取 direct/supporting/contradictory/missing；`hypothesis_critique` 读取 direct/supporting/contradictory；`runbook_retrieve` 和 `runbook_draft` 读取 direct/supporting/missing；`runbook_review` 读取 direct/contradictory/missing。`hypothesis_set.py` 已经进入代码。这个对象从 `InferenceResult + Evidence Pack V2` 生成结构化假设集。字段包括 `set_id`、`primary_hypothesis_id`、`items` 和 `summary`。每条 hypothesis item 当前固定为 `hypothesis_id`、`rank`、`statement`、`confidence_score`、`confidence_label`、`support_evidence_refs`、`contradict_evidence_refs`、`missing_evidence_refs`、`next_best_action` 和 `review_state`。`review_verdict.py` 已经进入代码。这个对象基于 `Evidence Pack V2`、`HypothesisSet` 和 `runbook_plan_outline` 生成结构化审查结果。字段包括 `verdict_id`、`verdict_status`、`recommended_disposition`、`approval_required`、`blocking_issues`、`checks` 和 `review_summary`。`checks` 当前固定覆盖 `evidence_sufficiency`、`temporal_freshness`、`topology_consistency`、`overreach_risk`、`remediation_executability` 和 `rollback_readiness`。`rule_based_seed_adapter.py` 是当前 bridge。输入是 alert、cluster trigger、`recent_similar_1h` 和 `history_support`。输出是 `reasoning_runtime_seed`。

## 机制

当前系统里的 downstream path 采用两段结构。第一段是 deterministic candidate graph construction。输入来自 alert contract、cluster aggregator、ClickHouse history lookup 和现有 context 字段。输出是一张带有候选节点和候选关系的局部事件图。第二段是 explanation-oriented reasoning。这个阶段当前还在 foundation 期。后续状态机将围绕 `ClusterSelectOrRefine -> ContextAssemble -> EvidencePackBuild -> HypothesisGenerate -> HypothesisCritique -> RunbookRetrieve -> RunbookDraft -> RunbookReview -> SuggestionProjection` 展开。状态机外部保留 `human_gate`。回退边保留 `cluster_too_broad`、`evidence_insufficient`、`hypothesis_conflict`、`runbook_not_applicable`、`runbook_overreach` 和 `high_risk_or_missing_approval`。

当前 context engineering 采用对象约束方式推进。`EvidencePack V2` 现在已经显式拆出 `direct_evidence`、`supporting_evidence`、`contradictory_evidence`、`missing_evidence`、`freshness`、`source_reliability` 和 `lineage`。phase router 按阶段分配这份对象的视图。`HypothesisGenerate` 当前读取 direct/supporting/contradictory/missing 证据，再读取 rule/path/device 上下文。`HypothesisCritique` 当前读取 direct/supporting/contradictory 证据，再读取 policy/sample/change 上下文。`RunbookRetrieve` 和 `RunbookDraft` 当前读取 direct/supporting/missing 证据，再读取 device/policy/change 和 `runbook_plan_outline`。`RunbookReview` 当前读取 direct/contradictory/missing 证据，再读取 change、历史摘要和审批边界。当前 suggestion payload 已经带上 `hypothesis_set` 和 `review_verdict`。前端 projector 的 inference / action 节点已优先读取这两个对象。旧 runtime suggestion 不带这两个对象时，gateway 会生成兼容 fallback。working memory 保留 `accepted_hypotheses`、`rejected_hypotheses`、`unresolved_questions`、`missing_evidence`、`pending_checks` 和 `risk_flags`。context compression 压缩结构化摘要。answer contract checker 在最终投影前检查证据引用、事实与推断分离、缺失信息、审批边界和回滚边界。

runbook 路径已在当前架构中预留正式位置。当前对象是 `runbook_plan_outline`。后续对象是 `RunbookPlan`。目标字段包括 applicability、evidence refs、prechecks、diagnostic steps、recommended actions、risk level、blast radius、approval required、rollback prerequisites、rollback steps 和 missing information。当前分支不引入自动执行。当前分支不引入设备写回。工具层保留 allowlist、参数校验、超时、结果结构化和调用审计。

## 模型服务接口

模型接入继续沿用现有 provider 抽象。`template` provider 保持当前路径。外部模型服务通过 `http`、`gpu_http` 或 `external_model_service` provider 标识接入。配置项已经在 `AgentConfig` 中显式化，包括 `provider_compute_target` 和 `provider_max_parallelism`。当前默认约束是 `provider_compute_target=external_gpu_service`、`provider_max_parallelism=1`，由核心节点发起请求。当前代码里 `provider_routing.py` 会生成结构化 routing hint，包含 `compute_target`、`max_parallelism`、`request_kind`、`suggestion_scope`、`candidate_event_graph_id`、`investigation_session_id` 和 `runbook_plan_id`。这个 hint 已经可以作为远端 GPU 服务的输入契约一部分。

边缘节点不承载模型执行。核心节点承载 suggestion、evidence、session 和 future orchestration。当前核心节点内存为 `14 GiB`。这一预算适合模板 provider、本地结构化对象构造、阶段感知上下文裁剪和远端模型转发。它不适合和当前 Kafka consumer、ClickHouse 查询、gateway 进程一起承载大模型。GraphRAG 目前只规划给非结构化知识面。运行时对象和当前热路径不依赖完整 GraphRAG indexing。

## 参数与评测

当前仓库里已经存在几组与这个 path 直接相关的参数。`AIOPS_CLUSTER_WINDOW_SEC`、`AIOPS_CLUSTER_MIN_ALERTS` 和 `AIOPS_CLUSTER_COOLDOWN_SEC` 控制 cluster trigger 的窗口、触发阈值和冷却时间。`AIOPS_PROVIDER`、`AIOPS_PROVIDER_ENDPOINT_URL`、`AIOPS_PROVIDER_MODEL`、`AIOPS_PROVIDER_TIMEOUT_SEC`、`AIOPS_PROVIDER_COMPUTE_TARGET` 和 `AIOPS_PROVIDER_MAX_PARALLELISM` 控制 provider 路径。`CLICKHOUSE_*` 参数控制历史检索来源。当前挂载 runtime 中有 `691` 个 alert JSONL 文件和 `201003` 条 alert，时间范围是 `2026-03-04T15:09:11+00:00` 到 `2026-04-02T16:23:04+00:00`。当前挂载 runtime 中有 `603` 个 suggestion JSONL 文件和 `222023` 条 suggestion，时间范围是 `2026-03-09T05:08:56.549849+00:00` 到 `2026-04-05T18:03:18.303384+00:00`。最近 24 个 alert 分桶中有 `1452` 条 `deny_burst_v1|warning` 和 `1` 条 `bytes_spike_v1|critical`。最近 24 个 suggestion 分桶中有 `1690` 条 `alert` 建议和 `32` 条 `cluster` 建议。

评测项按对象拆开定义。候选事件图阶段测 node count、edge count、relation type 分布和 seed coverage。`EvidencePack V2` 阶段测 direct/supporting/contradictory/missing evidence 覆盖率。hypothesis 阶段测 hypothesis 数量、冲突率和证据引用率。review 阶段测 overreach rejection rate、rollback readiness coverage 和 evidence sufficiency rejection rate。runbook 阶段测 applicability hit rate、precheck completeness 和 approval boundary coverage。trace 阶段测 trace completeness、tool call linkage 和 replay success rate。系统级对比实验保留 deterministic baseline、template suggestion baseline、typed evidence baseline、hypothesis critique loop 和 runbook planning loop。
